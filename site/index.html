rdbg

Debug a Rust program from a coding agent. Set breakpoints, run, and read the real values of your variables — instead of guessing from println! and recompiling.

# macOS and Linux
curl -fsSL https://azimi.me/rust-debugger-skill/install.sh | sh

A single static binary over rust-analyzer and lldb-dap. Use it as a skill (the rdbg CLI) or an MCP server (rdbg mcp) in Claude Code and Codex.

$ rdbg launch --cargo . --bin app --break src/config.rs:88
>>> STOP [breakpoint] app::parse_config  config.rs:88  (thread 1)
   ->    88 |     let cfg = Config::from(&raw);
$ rdbg vars
  cfg: Config
    threads: usize = 4
    paths: Vec<PathBuf> (size=2)
$ rdbg eval cfg.threads      # usize = 4
$ rdbg set cfg.threads = 8   # change it, keep running
$ rdbg step over

Or watch a value evolve across a whole loop in one call, instead of stepping and re-reading it each time:

$ rdbg trace --cargo . --bin app --break src/sum.rs:44 --capture i,sum
trace: 3 hit(s)
 #1   app::total  sum.rs:44   i=0  sum=0
 #2   app::total  sum.rs:44   i=1  sum=3
 #3   app::total  sum.rs:44   i=2  sum=3   <- sum stops advancing

Why

An agent editing Rust can read the source but not the run. It sees that parse_config exists; it can't see that threads came back 0. The usual workaround — add println!, rebuild, read the log, delete it — is slow and only shows what you thought to print. rdbg gives the agent the other half: break where a value is wrong, read it, step to watch it change, and change it in place to test a fix.

What it does

  • Breakpoints: line, function, conditional, hit-count, logpoint
  • Panic breakpoints — stop on the frame that raised a panic
  • Watchpoints — stop when a value changes
  • Step over / in / out / instruction, run-to-line, pause, restart
  • Locals as real Rust values (Vec, String, structs, enums)
  • Evaluate variable paths; set a variable live
  • Watch expressions, backtraces, thread & frame selection
  • Jump to definition, hover, references (rust-analyzer)

Use it as an MCP server

The same binary exposes 24 tools. Claude Code — .mcp.json, or claude mcp add rustdbg -- rdbg mcp:

{ "mcpServers": { "rustdbg": { "command": "rdbg", "args": ["mcp"] } } }

Codex — ~/.codex/config.toml:

[mcp_servers.rustdbg]
command = "rdbg"
args = ["mcp"]

How it works

A per-project daemon holds one paused lldb-dap session and a warm rust-analyzer, and serves commands over a Unix socket. The CLI and the MCP server are thin clients, so a breakpoint set in one call is still there in the next and the program stays paused between an agent's tool calls.

Requirements

  • rust-analyzer — rustup component add rust-analyzer
  • lldb-dap — Xcode command line tools on macOS, apt install lldb on Linux. codelldb, if present, adds full expression evaluation.
  • A debug build (the default cargo build).
